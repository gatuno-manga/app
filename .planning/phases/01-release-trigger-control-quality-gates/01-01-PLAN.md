  <action>Crie `.github/workflows/release.yml` seguindo o estilo existente de GitHub Actions do repositório. Configure `on.push.tags: ['v*']` e valide no primeiro job que `github.ref_type` é `tag` e `github.ref_name` respeita `^v(0|[1-9][0-9]*)\.(0|[1-9][0-9]*)\.(0|[1-9][0-9]*)$` (D-01). Em caso inválido, falhe imediatamente com mensagem clara antes de qualquer caminho de build/release (D-02). Configure `concurrency.group: ${{ github.workflow }}-${{ github.ref }}` com `cancel-in-progress: true` (D-03, D-04). No mesmo job de validação, faça checkout com histórico completo e valide ancestralidade da tag em `origin/main` via `git merge-base --is-ancestor` (D-09). Use permissões mínimas no workflow/job (`contents: read` nesta fase), preservando `contents: write` apenas para fases de publicação futura (D-08). Não implementar fallback "v1 simplificado": validação deve ser completa já nesta fase.</action>

    - Concurrency cancela execução anterior da mesma tag e mantém a mais nova.
    - Tag fora da ancestralidade de `main` falha na validação.
  </acceptance_criteria>
  <verify>
    <automated>grep -nE "tags:|concurrency:|cancel-in-progress: true|github\\.workflow.*github\\.ref|merge-base --is-ancestor|Invalid release tag" .github/workflows/release.yml</automated>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| maintainer→GitHub Actions trigger | Entrada não-confiável via nome da tag publicada |
| workflow runner→repository history | Validação de ancestralidade depende de estado git remoto (`origin/main`) |
| workflow jobs→GITHUB_TOKEN permissions | Escopo excessivo pode permitir ações além do necessário |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | T | `.github/workflows/release.yml` tag parser | mitigate | Validar `github.ref_type == tag` e regex estrita `vMAJOR.MINOR.PATCH` no primeiro job; falhar com exit 1 antes de qualquer downstream. |
| T-01-02 | D | Workflow run scheduler | mitigate | Configurar `concurrency` por `workflow+ref` com `cancel-in-progress: true` para impedir múltiplas execuções concorrentes por versão. |
| T-01-03 | E | `GITHUB_TOKEN` scope | mitigate | Definir permissões mínimas (`contents: read`) no job de validação; não conceder `contents: write` nesta fase. |
| T-01-04 | T | Release source integrity | mitigate | Verificar ancestralidade da tag em `origin/main` via `git merge-base --is-ancestor` e abortar quando não alcançável. |
</threat_model>
